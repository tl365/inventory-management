Frontend Patterns

Singleton Composable Filter Pattern

useFilters.js holds 4 reactive refs as a single source of truth shared across all views. Each view reads the same composable instance — no prop drilling or Vuex needed.

Raw Refs + Computed Derived Data

Raw API response stored in refs (allOrders, inventoryItems). Display-ready data derived in computed() — e.g. topProducts, orderHealthMetrics, statusData.

Centralized API Client

All endpoint calls live in api.js. Filter state from useFilters is mapped to URLSearchParams in one place, keeping views free of HTTP logic.

Watcher-Triggered Reloads

Each view uses watch([selectedPeriod, selectedLocation, selectedCategory, selectedStatus], loadData) to reactively refetch when any filter changes.

Client-Side Search

Inventory search filters the locally-held inventoryItems ref without an extra API call. Search input is not sent to the backend.

Custom SVG Charts

Donut charts and bar charts built with inline SVG and stroke-dasharray/stroke-dashoffset — no charting library dependency.

Modal-Based Detail Views

Detail content is surfaced via modals: ProductDetailModal, InventoryDetailModal, BacklogDetailModal. List rows emit events to open them.

i18n Composable

useI18n.js maps locale to currency automatically — en uses USD, ja uses JPY. Toggle stored in localStorage and reactive across components.

Backend Patterns

Pydantic Models for All Responses

Every FastAPI endpoint has a response_model annotation. Pydantic validates outgoing data, catching schema mismatches before they reach the client.

Utility Filter Functions

filter_by_month() handles monthly + quarterly periods (e.g. Q1-2025). apply_filters() handles warehouse, category, and status. Both skip when value is 'all'.

Computed Fields at Endpoint Level

has_purchase_order on BacklogItem is computed by cross-referencing the purchase_orders list at request time — no denormalized field in the JSON.

No Database, No ORM

All data loaded at startup from JSON into module-level variables in mock_data.py. Data is held in-memory for the process lifetime — restart to reset.

'all' Sentinel Value

When a filter query param is 'all', the filter is a no-op. This convention is explicit in both the frontend (api.js) and backend filter utilities.

Data Loaded Once at Import

mock_data.py reads JSON files and parses them into Python dicts/lists at module import time. Endpoint handlers reference these module-level variables directly.

Data Flow

1
User selects a filter FilterBar.vue renders the 4 dropdowns: Time Period, Warehouse, Category, Order Status.
2
Composable updates reactive ref useFilters.js exposes selectedPeriod, selectedLocation, selectedCategory, selectedStatus as shared reactive state.
3
View's watcher triggers loadData() Each view watches the 4 filter refs. Any change fires loadData(), which calls the appropriate api.js function.
4
api.js maps state to URLSearchParams Composable values are translated into query parameters (e.g. ?warehouse=tokyo&category=electronics&month=2025-01) and sent via axios.
5
FastAPI receives query params Route handler extracts typed params (e.g. warehouse: str = "all"). FastAPI validates types automatically.
6
apply_filters() + filter_by_month() process data In-memory lists from mock_data.py are filtered in Python. The 'all' sentinel skips each filter.
7
Pydantic validates response The response_model annotation serializes and validates the filtered data before it leaves FastAPI.
8
Vue stores in ref, computed derives display data Raw response stored (e.g. allOrders.value = data). Computed properties immediately recompute: topProducts, statusData, etc.
9
Template renders reactively Vue's reactivity system triggers a re-render wherever computed properties are used in the template. No manual DOM updates.
